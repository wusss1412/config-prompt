请按以下规范为我配置 Codex 的【用户全局层】定制。
这是一份 Codex 专用 config-prompt，可以直接提到 Codex、config.toml、AGENTS.md、hooks.json、my-global-config-prompt.md 等文件名。
不要在本文档正文中写入任何本机绝对路径；需要表达位置时使用文件名、相对概念或“Codex 用户全局配置目录”这类描述。
下面主要描述意图与效果；具体字段名和实现方式必须以当前 Codex 版本实际支持的配置格式为准。

═══════════════════════════════════════════════════════════
1. 权限模式
═══════════════════════════════════════════════════════════

默认审批策略保持“按需确认”，不要改成完全不确认。

允许在用户全局层启用更宽松的本地执行/文件访问能力，并关闭对应的高权限启动提示，避免每次开会话被无关提示打断。

同时必须保留一份“禁止执行”黑名单，显式拦截以下不可逆或高破坏性操作：

- 任意磁盘根或家目录的递归强删
- 卷格式化、修改本机执行策略
- 版本控制中的破坏性操作：强制推送、强制重置、强制清理工作区
- 直接管道执行远程脚本（把网络下载的内容直接喂给 shell）

效果：日常本地操作尽量少打断，但真正会丢数据、丢提交或改变系统状态的命令仍被硬拦。

═══════════════════════════════════════════════════════════
2. 默认模型与外观
═══════════════════════════════════════════════════════════

- 默认模型选当前可用的最高能力档。
- 主题选一个适合长时间盯屏的暗色方案，最好对色弱友好。
- 自动更新跟最新发布通道。
- 启用自动上下文压缩：逼近上限时由 Codex 自己截短，避免硬截断丢历史。

═══════════════════════════════════════════════════════════
3. 状态栏
═══════════════════════════════════════════════════════════

如果支持自定义状态栏，优先使用 Codex 当前版本官方支持的内置状态栏项目，显示：

- 当前模型名
- 当前工作目录或项目名
- 当前分支（如果可用）
- 上下文窗口占用率：10 格 ASCII 进度条 + 百分比，每填一个 # 约等于 10%（如果可用）
- 账号用量窗口剩余额度百分比和 reset 时间（如果可用）

只有在 Codex 当前版本明确支持“通过外部命令生成状态栏内容”时，才按平台选择脚本语言挂载自定义脚本。
如果 Codex 当前版本只支持内置项目列表，不要把 shell 命令、脚本路径或命令参数写进状态栏项目列表。
当前 Codex 版本的状态栏项目必须使用内置 ID，例如 model-with-reasoning、context-used、five-hour-limit、weekly-limit；不要使用 cwd、branch、tokens、primary-used-percent、secondary-used-percent 这类无效短名。
如果支持状态栏进度指示开关，启用它以显示上下文窗口占用率进度条；如果不支持上下文已使用百分比或 ASCII 进度条，使用最接近的内置 token/context 项目替代；若完全不支持自定义状态栏，明确说明跳过原因，不要强行模拟。

═══════════════════════════════════════════════════════════
4. 会话结束钩子：自动清理“垃圾会话”
═══════════════════════════════════════════════════════════

如果支持会话结束时触发钩子，注册一个钩子，只处理本次会话自己的记录文件。命中以下任一规则即删除该记录文件：

规则 A：用户根本没说过话

- 扫描所有“用户消息”
- 以尖括号开头的系统注入内容视为非真实用户输入
- 以 “Caveat:” 开头的内容视为非真实用户输入
- 如果真实用户消息数为 0，则删除

规则 B：用户最多说了一句

- 用户消息总条数小于或等于 1，则删除

硬约束：

- 钩子的任何异常都必须吞掉，绝不阻塞 Codex 退出
- 必须有几秒级兜底超时，防止卡死
- 删除与异常都要追加写日志，方便事后审计
- 只处理本会话自己的记录，不追溯扫描其他历史
- 该自动清理行为不需要删除前再向用户确认

效果：会话恢复/历史列表不再被误开关、单字符回复、纯命令调用这类垃圾条目刷屏；实际写过内容的会话一条都不应丢。

═══════════════════════════════════════════════════════════
5. 长期记忆偏好
═══════════════════════════════════════════════════════════

如果支持长期跨会话记忆，维护以下偏好规则：

(1) 输出文件后必须报告绝对路径

每次写/改文件之后，回复里都要明确列出落盘的绝对路径；一次改多个文件时用表格列出，不要让用户去猜。

原因：用户要立刻能跳到改动位置去复核。

(2) 删除前必须列清单 + 显式确认

删除任何文件之前，先把要删的清单列出来，等用户明确同意后再动手。

例外：用户全局会话清理钩子删除命中的垃圾会话记录时，不需要额外确认。

原因：避免误删用户在做的工作，同时允许已定义的自动清理机制正常工作。

(3) 用户全局配置说明必须同步维护

每当进行“用户全局”“全局配置”这类定制修改时，同步更新这份说明文档，使文档描述和当前实际配置保持一致。

凡是修改实际全局配置、全局钩子、全局提示词、长期偏好或用户全局层相关文件，都必须把“检查这份说明文档是否匹配当前设置”作为完成条件。
如果不匹配，必须同步更新这份说明文档；如果已经匹配，最终回复也要明确说明“已检查，无需更新”。
最终回复必须同时列出本次修改过的实际配置文件，以及这份说明文档是否被修改或已检查。

文档内可以出现 Codex 和相关配置文件名，但不要出现任何本机绝对路径。
描述应尽量保持可迁移：优先写意图、效果、约束和当前 Codex 版本需要遵守的配置格式。

原因：用户希望这份文档能作为后续重建 Codex 全局配置的提示，同时避免绑定某台机器的绝对路径。

(4) DOCX 学术文档修改必须保守处理

当修改 Word `.docx` 论文、参考文献、交叉引用、公式或图表时，必须把 `.docx` 当作 ZIP 包处理，并优先使用 `zipfile` 与 `lxml.etree` 检查 `word/document.xml`。

参考文献或引用编号调整必须以用户给出的原始 `.docx` 为正文底稿，不得重建、重排或搬移正文段落；只允许替换参考文献区域，并更新现有 `REF _Ref...` 域、显示编号与书签目标。

公式段落不得扁平化为普通文本，因为公式参数可能由 Word run、嵌入对象、上下标或特殊格式组成。完成前必须对比原始文件在“参考文献”标题之前的正文段落顺序、正文文本、段落属性、对象/图片/上下标/制表符/换行/节属性，并验证所有引用域目标和书签一一匹配。

如果发现章节内容错位、公式参数变成普通文本、引用显示正确但跳转目标错误等现象，必须停止基于坏文件继续补丁，改为从原始 `.docx` 重新构建最小修改版本。

原因：避免论文正文被重排、公式结构被破坏、交叉引用跳转失效。

(5) DOCX 错误日志必须沉淀到全局规则

每当 DOCX 修改中出现可复现的错误，例如整段替换导致 run 级格式被压平、上下标丢失、交叉引用域变成普通文本、删图后正文仍残留图号引用、插图后媒体关系未校验、只修可见问题但漏掉全文连续性检查，都必须把错误原因和对应反制措施记录到用户全局 `AGENTS.md`。

记录时不要只写“使用 zipfile 和 lxml”，还必须写清楚保守编辑策略，例如是否保留原始 `w:r`、`w:rPr`、`w:fldChar`、`w:instrText`、`w:vertAlign`，以及新增内容是否应作为独立段落插入。

原因：把实际踩过的坑转化为后续会话的硬约束，避免同一类论文格式错误反复出现。

(6) Skills 自动加载范围必须保持精简

Codex 用户全局 `skills` 根目录只保留近期高频使用或基础能力相关的技能，例如系统内置技能、Office 文档处理、引用检索、PPT/幻灯片、PDF、表格和网页测试类技能。

批量迁移、低频或领域专用技能不要直接堆在自动加载根目录；应保存在旁边的 disabled/归档目录中，需要使用时再移回 `skills` 根目录。

可以保留很小的自动加载路由技能，例如 `skill-index`，以及按能力域组织的分组技能，例如 `gsd`。同一体系的大量子技能应放在分组技能内部的 `library/` 中，由该分组技能按需读取具体子技能的 `SKILL.md`；不要把同一体系的大量技能平铺在根级 `skills` 目录，也不要优先依赖 `skills.disabled` 下的分组索引。只有尚未归入分组技能的低频技能，才通过 disabled 总索引按需恢复单个技能。

原因：根目录技能过多会挤占 skills 描述预算，触发“Skill descriptions were shortened to fit the 2% skills context budget”警告，并降低技能触发描述的可读性。

═══════════════════════════════════════════════════════════
6. 验证
═══════════════════════════════════════════════════════════

- 配置文件必须能被 Codex 正常加载
- 黑名单内的破坏命令应被拦截
- 普通日常命令不应被黑名单误拦
- 状态栏装好后底部应显示模型名、上下文窗口占用率进度条和百分比、账号 5 小时/周窗口剩余额度百分比
- 新开空会话后退出，下一次查看历史列表不应看到该垃圾会话；清理日志应记录删除原因
- 后续任何一次写文件操作，回复里应自动带绝对路径
- 新会话的 skills 列表不应再触发 2% skills context budget 描述截短警告
